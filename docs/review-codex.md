## 주요 발견사항 (심각도순)

### 🔴 높음

**1. `search_similar_content` 재정의가 기존 데이터 소스를 제거함**
- `003`에서 함수를 `CREATE OR REPLACE`로 덮어쓰면서 `chat_messages` 검색이 빠지고 `telegram_messages`만 포함됨
- 기존 기능 회귀 가능성이 큼

**2. 임베딩 전략이 문서 내에서 상충됨 (Gemini 전용 vs Fireworks 폴백)**
- 핵심 원칙은 Gemini 임베딩 전용인데, 검증 포인트는 Fireworks 폴백을 전제
- 모델 혼용 시 벡터 공간 불일치로 검색 품질이 깨질 수 있음

**3. `pending_messages.json` 기반 큐에 동시성 제어 명세 없음**
- 리스너/체커/실행기가 동시에 접근하는데 파일 락/원자적 write/복구 규칙 없음
- 메시지 유실·중복·JSON 손상 위험이 큼

**4. `--dangerously-skip-permissions` 사용이 보안 경계 자체를 약화**
- CLI가 파일/환경변수 접근 제한 없이 동작할 수 있어 비밀정보(.env, 토큰) 노출면이 커짐
- 화이트리스트만으로는 방어가 불충분

### 🟡 중간

**5. 10단계 워크플로우의 식별자 계약 불명확 (msg_id 타입/원천)**
- `save_classification(msg_id, json)`에서 `msg_id`가 UUID(`telegram_messages.id`)인지 BIGINT(`telegram_message_id`)인지 모호
- 저장 실패/오기록 위험

**6. "8개 Python 파일" 설명이 실제 목록과 불일치**
- 목록은 Python 6개 + `executor.sh` + `CLAUDE.md`
- 의존성 검토 범위 정의 자체가 모순

**7. 폴링(10초)+실행 타이머(1분) 구조는 지연 상한이 큼**
- 최악 응답 지연이 약 70초(+실행시간)까지 늘어날 수 있음
- 실행 중 유입 메시지 처리 정책이 불명확

**8. TIMESTAMP(timezone 없음) 사용으로 시간 정합성 리스크**
- Python/Next.js/DB 타임존이 다르면 기간 조회·정렬·리포트 경계에서 불일치 가능

### 🟢 낮음

**9. 마이그레이션 의존성 명시 부족 (categories 선행 필요)**
- `telegram_messages`/`todos`가 `categories(id)` FK를 참조하므로 적용 순서 보장 필요

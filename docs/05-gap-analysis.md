# 갭 분석: 초기 플랜 vs 실제 요구사항

## 초기 플랜이 커버한 것

| 기능 | 상태 | 비고 |
|------|------|------|
| 채팅형 UI (다크모드) | ✅ 포함 | shadcn/ui + 사이드바 + 메시지 버블 |
| 자동 분류 + AI 조언 | ✅ 포함 | 메타데이터 카드로 표시 |
| 벡터 검색 장기 기억 | ✅ 포함 | pgvector + Gemini 임베딩 |
| 대화 세션 관리 | ✅ 포함 | 세션 목록, 생성, 삭제 |
| 스트리밍 응답 | ✅ 포함 | SSE 방식 |
| 기존 데이터 백필 | ✅ 포함 | /api/backfill |

| PC 활동 자동 추적 | ❌ 누락 | 시간 추적의 핵심 |
| 오프라인 활동 기록 | ❌ 누락 | |

| 시스템 범위 | 채팅 UI만 | 채팅 + 시간추적 + (에이전트) |

## 새 플랜에서 다뤄야 할 것

1. **모듈 구조**: 채팅/시간추적/에이전트를 어떻게 하나의 앱에 통합?
2. **Claude CLI 통합**: Next.js에서 child_process로 호출하는 안정적 방법
3. **활동 추적 데몬**: 크론잡 또는 상시 프로세스로 윈도우 기록
4. **DB 스키마 확장**: activity_logs, hourly_summaries, todos, fixed_schedules
5. **UI 확장**: Daily Report 페이지, 할일 관리, 오프라인 입력
6. **에이전틱 아키텍처**: Function Calling / Tool Use 설계 (Phase 2라도 구조는 잡아야)

## 구현 순서 제안 (재검토 필요)

### Phase 1A: 채팅 비서 (기반)
- DB 마이그레이션
- Claude CLI 통합
- 채팅 UI + 스트리밍
- 벡터 검색 기억

### Phase 1B: 시간 추적
- PC 활동 추적 데몬
- Daily Report UI (24시간 그리드)
- 할일 리스트
- AI 몰입도 판정

### Phase 2: 에이전트
- Tool/Function 아키텍처
- 기본 도구: 웹 검색, 일정 추가, 알림
- 승인 플로우 (AI 제안 → 사용자 확인 → 실행)

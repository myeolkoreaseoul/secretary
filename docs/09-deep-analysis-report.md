## Part 1: 소놀봇이 Claude CLI Max를 쓰는 방식 (완전 해부)

### 핵심 명령어

```batch
claude -p -c --dangerously-skip-permissions --append-system-prompt-file CLAUDE.md "텔레그램 메시지를 확인하고 처리할 것..."
```

| 플래그 | 의미 | 왜 필요한가 |
|--------|------|------------|
| `-p` | Print mode (비대화형) | 사람 입력 없이 실행 후 종료 |
| `-c` | Continue (이전 세션 이어서) | 이전 대화 맥락 전체 유지 |
| `--dangerously-skip-permissions` | 모든 도구 허가 건너뜀 | Python 실행, 파일 읽기/쓰기를 확인 없이 자동 수행 |
| `--append-system-prompt-file` | CLAUDE.md를 시스템 프롬프트로 추가 | 800줄짜리 지시서 (API 레퍼런스 + 워크플로우) |
| 마지막 문자열 | 유저 프롬프트 | "이 순서로 함수를 호출하라" 10단계 지시 |

### 작동 원리 (시퀀스)

```
Windows Task Scheduler (1분마다)
    ↓
mybot_autoexecutor.bat
    ↓
[안전 체크 4단계]
  1. tasklist + wmic → Claude 프로세스 이미 실행중? → 스킵
  2. lock 파일 있는데 프로세스 없음? → 스테일, 락 삭제 후 복구
  3. quick_check.py (0.1초) → 새 메시지 없음? → 즉시 종료
  4. lock 파일 생성
    ↓
claude -p -c --dangerously-skip-permissions ...
    ↓
Claude Code가 Bash 도구로 Python 실행:
  python -c "from telegram_bot import check_telegram; print(check_telegram())"
    ↓
10단계 워크플로우 자동 수행
    ↓
작업 완료 후 3분 대기 → 재확인 루프
    ↓
lock 파일 삭제
```

### Claude가 Python 함수를 호출하는 방법

Claude Code는 **Bash 도구**를 사용합니다. CLAUDE.md에 정의된 API를 보고:

```python
# Claude가 실제로 실행하는 코드 (Bash 도구 경유)
python -c "
from telegram_bot import check_telegram
result = check_telegram()
print(result)
"
```

즉, **Claude Code가 알아서 Python 코드를 작성해서 실행**합니다.
시스템 프롬프트(CLAUDE.md)에 함수 시그니처와 사용법이 적혀있으니까.

### 세션 연속성 (-c 플래그)

```
1분차: claude -p -c → 세션 시작, 작업 수행, 종료
2분차: claude -p -c → 같은 세션 이어서, 이전 맥락 전부 기억
3분차: claude -p -c → 계속 이어감
```

- `-c`는 가장 최근 세션을 자동으로 resume
- 이전에 어떤 파일을 읽었는지, 어떤 대화를 했는지 전부 기억
- 실패하면 `-c` 없이 새 세션으로 폴백

### 2-2. 채널 (텔레그램 vs 웹 UI)

| | 텔레그램 | 별도 웹 UI |
|---|---------|-----------|
| 개발 비용 | $0 (소놀봇 코드 그대로) | 높음 (8개 컴포넌트) |
| 접근성 | 폰+PC 어디서든 | 브라우저 필요 |
| 알림 | 푸시 알림 내장 | 직접 구현 필요 |
| 파일 전송 | 내장 (사진/문서/음성) | 직접 구현 |
| 위치 공유 | 내장 | 직접 구현 |
| 응답 속도 | 1분 폴링 지연 | 실시간 가능 |
| 커스텀 UI | 제한적 (텔레그램 포맷) | 무한 자유 |
| Daily Report | 텍스트만 | 그리드/차트 가능 |
| 시간 추적 UI | 불가 (24시간 그리드 안됨) | ✅ 가능 |

**선택: 텔레그램(입력) + 웹(기록/분석/현황 관리)**

**역할 분담:**
- 텔레그램 = **입** (말 거는 곳)
  - 이미 만들어진 앱, 알림, 파일, 위치, 어디서든 접근
  - 소놀봇 코드 재활용 → 개발 거의 불필요
- 웹사이트 = **두뇌의 정리된 기억** (보고 관리하는 곳)
  - 단순 대시보드가 아니라 **카테고리별 지식 관리 시스템**

```
텔레그램 (입력 — 매일 수시로):
  "개발 공부해야 한다" → 자동 분류 + 조언
  "비행기 예약해줘" → 에이전틱 실행
  "12시~1시 점심 먹었어" → 오프라인 활동 기록
  "김대리랑 미팅했어" → [업무]로 저장 + 조언

웹사이트 (관리/분석 — 하루 1~2번):
  /categories → 카테고리별 현황 (핵심!)
    [개발] 탭: AI 에이전트 공부(2/16), OpenClaw 분석(2/16)...
           → AI 요약: "AI 에이전트 개발에 집중 중. 진척률 약 40%"
    [소개팅비즈니스] 탭: 수익모델 고민(2/16)...
           → AI 요약: "수익모델 미확정. 구독제 vs 건당과금 검토 필요"
    [업무] 탭: 김대리 미팅(2/17)...
           → AI 요약: "이번 주 미팅 2건. 후속 조치 1건 미완료"
  /history → 전체 대화 타임라인 (검색 가능)
  /time → 24시간 Daily Report 그리드
  /todos → 할일 관리
  /settings → 고정 일정, 앱 분류 설정
```

**문제점:**
- 웹사이트 개발 비중이 커짐 (카테고리별 현황 페이지 추가)
- 텔레그램 대화 → 웹 데이터 동기화 필요
- 1분 폴링 지연 (실시간이 아님)

**해결방안:**
- 웹은 Next.js 기존 프로젝트 확장 (이미 셋업 됨)
- 동기화: 같은 PC의 같은 파일/DB를 읽으므로 자동
- 폴링 10초면 체감 충분
- 카테고리별 AI 요약은 하루 1회 또는 요청 시 Claude CLI로 생성

### 2-3. 메모리

| | 소놀봇 | OpenClaw | 선택 |
|---|--------|----------|------|
| 방식 | 파일 (keyword 검색) | 플러그인 | **파일 + Claude 직접 판단** |
| 검색 | 정확한 키워드만 | 설정에 따라 | Claude가 의미 이해 |
| 예시 | "AI 검색" → "AI" 글자 포함만 | - | "AI" → OpenClaw, Secretary 등 관련 전부 |

**최종 결정: 소놀봇식 파일 메모리 + Claude가 직접 읽고 판단**

**이유:**
- Claude Opus 4.6은 한국어 의미를 이해함 → 임베딩 없이도 관련 기록 판단 가능
  - "AI 관련 뭐 했지?" → "OpenClaw 분석", "Secretary 개발" 등 의미적으로 연결
  - 소놀봇 keyword 검색은 정확한 글자만 찾음 → 이건 부족
- 임베딩(Gemini) + 벡터DB(pgvector)를 추가하면 의존성/복잡도 2배 → Phase 1에서는 과도
- 개인 비서 하루 3~5건이면 1년 넘게 파일 메모리로 충분 (~1000건 이하)

**구현 방법:**
```
1. 소놀봇 구조 유지: tasks/msg_{id}/task_info.txt에 기록 저장
2. 카테고리별 폴더 분리 → "AI 관련" 물으면 [개발] 폴더만 읽기
3. Claude가 메모리 파일을 직접 읽고 관련 내용 판단
4. 24시간 대화 컨텍스트도 함께 유지 (소놀봇 기존 기능)
```

**문제점:**
- 파일 1000건+ 넘으면 Claude가 전부 읽는 데 시간 걸림 + 토큰 한도
- keyword 검색은 의미 검색보다 빈틈 있음 (1차 필터로는 OK)

**해결방안:**
- 카테고리별 폴더 분리로 검색 범위 축소
- 최근 N개만 읽기 + 키워드 1차 필터 → Claude 2차 의미 판단 (2단계 검색)
- 웹사이트의 카테고리별 현황도 같은 파일 메모리 기반으로 구현

> **📌 임베딩 제공자 최종 확정:**
> **Gemini text-embedding-004 단독 사용**
> - 무료 (1,000회/일), 768차원, 한국어 품질 양호
> - Supabase Pro pgvector와 조합 → 추가 비용 $0
> - Ollama: 2009 맥북(Core 2 Duo)에서 AVX 미지원 → 실행 불가로 탈락
> - Fireworks/Voyage 등: 종량제 → 원칙 위반으로 탈락
> - 한도 문제 시: 큐에 넣고 나중에 재시도 (하루 3~5건이면 한도 안 넘음)
> 파일 메모리는 병행 유지 (소놀봇 호환 + 백업용)
>
> **서버 환경 확정:**
> - 상시 서버: MacBook Pro A1278 (2009 Mid, Core 2 Duo, SSD, 8GB RAM, Ubuntu + OpenClaw 설치 완료)
> - 개발: 현재 노트북 (Ryzen 5 7535HS) → git으로 맥북에 배포
> - 맥북 역할: 텔레그램 폴링 + Claude CLI 호출 + 크론잡 → 중계/실행만

### 2-4. 에이전틱

| | 소놀봇 | OpenClaw | 선택 |
|---|--------|----------|------|
| 방식 | Claude가 직접 셸/파일 조작 | 25+ 내장 도구 | **소놀봇 방식 확장** |
| 브라우저 | Playwright 가능 (포그라운드) | 내장 | Playwright |
| 스킬 | 없음 | 100+ AgentSkills | 직접 추가 |

**선택: 소놀봇 방식 (Claude가 직접 실행) + 도구 확장**

**이유:**
- 소놀봇의 `--dangerously-skip-permissions`로 Claude는 이미 **뭐든 할 수 있음**
- 파일 읽기/쓰기, 셸 명령, Python 실행 전부 가능
- OpenClaw의 도구 시스템은 결국 Claude가 직접 하는 것과 동일한 결과
- 별도 Gateway 없이 더 심플

**브라우저 자동화 가능 여부:**
```
소놀봇 포그라운드 모드 (register_scheduler_FOREGROUND.bat):
  /RL HIGHEST (최고 권한) + /IT (인터랙티브) + /RU (현재 유저)
  → Chrome, Playwright, Selenium 실행 가능
  → Claude가 "pip install playwright && python -c '...'" 실행 가능
```

**즉, Claude에게 "스카이스캐너에서 2월 항공편 검색해줘"라고 하면:**
1. Claude가 Playwright 코드를 작성
2. Bash 도구로 실행
3. 결과 스크린샷 찍어서 분석
4. 텔레그램으로 결과 전송

**문제점:**
- `--dangerously-skip-permissions`는 위험 (악성 지시에 취약)
- Claude가 잘못된 명령을 실행할 수 있음 (rm -rf 등)
- Playwright 설치/관리가 복잡할 수 있음
- 매번 코드를 새로 짜야 함 (같은 작업도 매번 처음부터)

**해결방안:**
- 텔레그램 화이트리스트 (본인만 접근)
- CLAUDE.md에 금지 명령 명시 (시스템 파일 삭제 금지 등)
- 위험한 작업은 실행 전 텔레그램으로 확인 요청
- 자주 쓰는 작업은 Python 스크립트로 미리 만들어서 CLAUDE.md에 등록
- 로그인 필요 사이트 → 브라우저 프로필 지정, 기존 세션 재활용
- CAPTCHA → 한계. 수동 개입 필요 시 텔레그램 알림

**최종 결정:**
- Phase 1: Claude가 직접 코드 짜서 실행 (소놀봇 방식)
- 자주 쓰는 작업은 스크립트로 미리 준비
- Phase 2 필요 시: OpenClaw 스킬 형태로 정형화 검토

### 2-5. 자동 분류

| | 소놀봇 | OpenClaw | 선택 |
|---|--------|----------|------|
| 구현 | 없음 | 없음 | **직접 구현** |

**선택: Claude가 대화하면서 동시에 분류 + 조언**

**구현 방법:**
CLAUDE.md에 분류 규칙 추가. Claude가 응답할 때 대화 + 분류 + 조언을 한 번에 처리.

**카테고리 운영:**
- 초기 7개: 업무/소개팅비즈니스/온라인판매/건강/가족/개발/기타
- **자동 확장**: 기존 카테고리에 안 맞으면 Claude가 새 카테고리 자동 생성 (승인 불필요)
- 나중에 웹 /settings에서 정리 (병합, 삭제, 이름 변경)
- DB `categories` 테이블에 행 추가/수정

**응답 예시:**
```
유저: "강남 아파트 시세 좀 알아봐줘"

Claude: (기존 7개에 안 맞음 → [부동산] 자동 생성)

### 문제 1: 소놀봇 Windows → Ubuntu 포팅

**문제:** .bat, Task Scheduler, tasklist/wmic 전부 Windows 전용. "그대로 복사" 불가.

**해법: systemd (Linux 프로세스 관리 업계 표준)**

```
Windows                    → Ubuntu (systemd)
─────────────────────────────────────────────
Task Scheduler (5분마다)   → systemd timer
.bat 실행기                → .sh + systemd service
tasklist / wmic            → pgrep / pidof
파일 락                    → flock (Linux 표준)
```

구체적 파일:
```
/etc/systemd/system/secretary-bot.service      ← 텔레그램 봇 상시 실행
/etc/systemd/system/secretary-executor.service  ← Claude CLI 실행기
/etc/systemd/system/secretary-executor.timer    ← 1분마다 트리거
```

systemd 제공 기능: 크래시 자동 재시작, 부팅 시 자동 시작, 로그 관리(journalctl)

**검증:** ✅ 모든 Linux 서버가 이렇게 운영됨.

### 문제 2: secretary-meta JSON 안정성

**문제:** Claude 응답 안에 JSON 끼워넣으면 빠뜨리거나 포맷 깨질 수 있음.

**해법: 분류를 별도 단계로 분리 (업계 표준 패턴)**

```
Step 1: Claude → 유저에게 답변 (자연어만) → 즉시 텔레그램 전송
Step 2: 같은 메시지를 별도 호출로 분류 → "JSON만 출력해라" → DB 저장
```

장점:
- 답변과 분류가 섞이지 않음
- JSON 파싱 실패해도 답변은 이미 전송됨
- 분류 실패 시 재시도 가능
- Claude가 분류 → 품질 최상, 새 카테고리 판단도 정확

Claude CLI 2번 호출(답변+분류)에 대해:
- Max 구독은 무제한 → 호출 횟수 아끼려고 Gemini 위탁하면 안정성만 낮아짐
- Gemini는 임베딩 전용. 분류/답변/조언 모든 판단은 Claude만 담당.

**검증:** ✅ "LLM 구조화 데이터 추출" 별도 호출은 업계 정석.

### 문제 5: Gemini 무료 임베딩 지속성

**문제:** 구글이 무료 한도를 계속 축소 중.

**해법: 추상화 레이어 + Fireworks 폴백**

```python
async def generate_embedding(text):
    try:
        return await gemini_embedding(text)     # 주: Gemini 무료
    except QuotaError:
        return await fireworks_embedding(text)   # 백업: $0.008/1M
```

- 임베딩 함수를 하나의 인터페이스로 감싸고 내부에서 제공자 전환
- Gemini 한도 초과 시 Fireworks 자동 전환 (월 ~$0.31)
- 나중에 다른 제공자 추가도 코드 몇 줄

**검증:** ✅ 폴백 패턴은 업계 표준.

### 문제 6: 시간 추적 아키텍처 (설계 누락 발견)

**문제:** 시간 추적은 유저의 노트북 화면을 봐야 하는데, 서버(맥북)에서는 못 봄.

**해법: 노트북에 경량 수집기 + Supabase 직접 기록**

```
Ryzen 노트북 (Windows):
  Windows Task Scheduler → 1분마다 PowerShell 실행
  → 활성 윈도우 제목 캡처
  → Supabase REST API로 직접 INSERT (HTTPS)

맥북 (Ubuntu):
  → 텔레그램 봇, Claude CLI, 웹 대시보드
  → 매시간 DB에서 활동 데이터 집계
  → 매일 Claude로 Daily Report 생성
```

노트북 스크립트 = PowerShell 1개 파일 (30줄 이내)
Windows Task Scheduler에 등록 → 부팅 시 자동 실행

**검증:** ✅ PowerShell + Task Scheduler = Windows 표준. Supabase REST API = HTTPS POST.

### 문제 7: 웹 대시보드 외부 접근

**문제:** 맥북이 집 안에 있으면 밖에서 웹 대시보드 못 봄.

**해법: Cloudflare Tunnel (무료)**

```
맥북(집) → Cloudflare Tunnel → https://secretary.도메인.com → 어디서든 접속
```

- 포트포워딩 불필요
- HTTPS 자동
- DDoS 방어 내장
- 설치: cloudflared 설치 → tunnel 생성 → DNS 연결 → systemd 등록

비교:
| 방법 | 비용 | 안정성 |
|------|------|--------|
| **Cloudflare Tunnel** | 무료 | ✅ 매우 안정 |
| Tailscale | 무료 | ✅ 안정 (VPN) |
| ngrok | 무료 제한 | ⚠️ URL 바뀜 |
| 포트포워딩 | 무료 | ⚠️ 보안 취약 |

**검증:** ✅ 수십만 명이 홈서버에 사용. 1인 개발자 표준.

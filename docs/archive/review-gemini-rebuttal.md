### **구현 전 반드시 수정해야 할 TOP 5**

Codex의 분석과 Gemini의 관점을 종합하여, 시스템의 안정성, 데이터 무결성, 보안에 가장 치명적인 영향을 미치는 5가지 항목을 선정했습니다.

**1. 임베딩 전략의 모순 및 단일화**
*   **문제점:** Gemini와 Fireworks 임베딩 모델을 혼용하려는 계획은 벡터 공간을 깨뜨려 의미 기반 검색의 정확성을 완전히 파괴합니다.
*   **위험:** 관련 콘텐츠 검색 기능이 전혀 작동하지 않거나, 완전히 엉뚱한 결과를 반환하게 됩니다.
*   **해결 방향:** **하나의 임베딩 모델(예: Gemini)을 선택하고 모든 임베딩 생성 및 검색 과정에서 일관되게 사용**하도록 아키텍처를 수정해야 합니다. 폴백(fallback) 전략이 필요하다면, 임베딩 모델이 아닌 서비스 엔드포인트 자체의 가용성에 대한 폴백이어야 합니다.

**2. 파일 기반 큐(`pending_messages.json`)의 동시성 제어**
*   **문제점:** 여러 프로세스(리스너, 체커, 실행기)가 원자적(atomic) 연산이나 잠금(locking) 메커니즘 없이 JSON 파일에 동시에 접근하고 수정합니다.
*   **위험:** 메시지 처리 누락, 중복 처리, 파일 깨짐(JSON 구조 손상)이 발생하여 데이터 유실로 이어집니다.
*   **해결 방향:** 파일 수정 시 **`flock`과 같은 파일 시스템 레벨의 잠금을 사용**하거나, 더 안정적인 **Redis 리스트나 PostgreSQL 같은 트랜잭션을 지원하는 DB를 큐로 사용**하는 것을 강력히 권장합니다.

**3. `search_similar_content` 함수의 회귀 버그 수정**
*   **문제점:** `CREATE OR REPLACE FUNCTION` 구문으로 인해 기존 `chat_messages` 테이블을 검색하는 로직이 삭제되고 `telegram_messages`만 검색하게 됩니다.
*   **위험:** 기존에 잘 작동하던 채팅 메시지 기반의 콘텐츠 검색 기능이 예고 없이 사라지는 심각한 기능 회귀가 발생합니다.
*   **해결 방향:** 함수를 수정하여 **`chat_messages`와 `telegram_messages` 모두를 `UNION ALL`로 묶어 검색**하도록 로직을 즉시 복원해야 합니다.

**4. 데이터 타입 및 식별자(ID) 명세 통일**
*   **문제점:** `msg_id`가 UUID인지 BIGINT인지 불명확하고, `TIMESTAMP`에 타임존이 없어 컴포넌트 간 데이터 정합성이 깨질 위험이 큽니다.
*   **위험:** FK 제약조건 위반으로 인한 DB 쓰기 실패, 잘못된 데이터 연결, 시간대 차이로 인한 리포트 집계 오류 등 예측 불가능한 버그가 발생합니다.
*   **해결 방향:** **모든 ID의 타입(예: `msg_id`는 UUID)과 출처를 명확히 정의**하고, 모든 타임스탬프 컬럼은 **`TIMESTAMPTZ` (timestamp with time zone) 타입으로 변경**해야 합니다.

**5. CLI의 과도한 권한 문제 해결**
*   **문제점:** `--dangerously-skip-permissions` 플래그는 CLI에 무제한적인 파일 시스템 및 환경 접근 권한을 부여하여 보안을 취약하게 만듭니다.
*   **위험:** 악의적인 사용이나 스크립트 오류 시, `.env` 파일의 API 키나 개인 설정 파일 등 민감 정보가 유출될 수 있습니다.
*   **해결 방향:** 해당 플래그 사용을 지양하고, **CLI가 접근해야 할 최소한의 파일 및 디렉터리 경로를 명시적으로 지정하여 전달**하는 방식으로 변경해야 합니다. (Whitelist-based approach)
